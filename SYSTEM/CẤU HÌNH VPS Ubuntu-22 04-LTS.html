4) DirectAdmin: cài đặt (VPS đã có sẵn)

4.1) Yêu cầu

  • Có license DirectAdmin (Client ID + License ID) hoặc key
  • VPS sạch, không cài sẵn web stack lẫn lộn

4.2) Cài đặt (cách phổ biến)

Tải script cài đặt (theo hướng dẫn của DirectAdmin):

cd /root
wget -O setup.sh https://www.directadmin.com/setup.sh
chmod 755 setup.sh
./setup.sh

Trong quá trình cài:

  • Chọn web server: OpenLiteSpeed (nếu menu cho chọn)
  • Chọn DB (thường MariaDB)
  • Chọn email stack nếu cần

Sau khi xong, truy cập:

  • http://<IP-VPS>:2222
  • hoặc https://<IP-VPS>:2222 (nếu đã có SSL cho panel)

5) Bật OpenLiteSpeed trong DirectAdmin (CustomBuild) (VPS hỗ trỡ sẵn)

Trên DirectAdmin, web stack thường quản lý qua CustomBuild.

Vào thư mục custombuild:

cd /usr/local/directadmin/custombuild
./build update
./build version

Thiết lập dùng OpenLiteSpeed (tuỳ bản DA, có thể là openlitespeed hoặc ols):

./build set webserver openlitespeed
./build set php1_mode lsphp
./build set php2_mode lsphp

Build lại:

./build webserver
./build php n
./build rewrite_confs

Kiểm tra service:

systemctl status directadmin
systemctl status lsws

6) SSL/TLS với Let’s Encrypt (khuyến nghị làm trong DirectAdmin)

Cách đơn giản và ít lỗi nhất là bật Let’s Encrypt trong DirectAdmin:

  • Admin Level → Server Manager → Let’s Encrypt
  • Bật Let’s Encrypt cho user/domain

Ghi chú quan trọng:

  • Domain phải trỏ đúng A record về VPS

Nếu muốn kiểm tra nhanh DNS:

# Trả về IP VPS là đúng
nslookup yourdomain.com

Tuỳ nhu cầu, thường có 2 hướng:

7.3) Deploy Java WAR (Java 17) với WildFly + reverse proxy qua OpenLiteSpeed

Mục tiêu:

7.3.1) Cài Java 17 (khuyến nghị OpenJDK) không dùng yum, cài bằng tar.gz

Trên CentOS 7, để không phụ thuộc yum/repo (dễ thiếu gói hoặc lệch repo), bạn có thể cài Eclipse Adoptium Temurin 17 bằng file tar.gz.

1) Tạo thư mục chứa Java

mkdir -p /usr/java
cd /usr/java

2) Download JDK 17 (Temurin)

Có 2 cách:

  • Cách A (khuyến nghị): lấy link release cụ thể (ổn định, ít lỗi 404)

    1) Mở trang release: https://github.com/adoptium/temurin17-binaries/releases

    2) Chọn bản mới nhất, copy link file dạng OpenJDK17U-jdk_x64_linux_hotspot_*.tar.gz

    3) Download:

wget -O OpenJDK17U-jdk_x64_linux_hotspot.tar.gz "<PASTE_LINK_TAR_GZ_HERE>"

Vi du: wget -O OpenJDK17U-jdk_x64_linux_hotspot.tar.gz https://github.com/adoptium/temurin17-binaries/releases/download/jdk-17.0.9%2B9/OpenJDK17U-jdk_x64_linux_hotspot_17.0.9_9.tar.gz
  • Cách B (nhanh): dùng “latest” (có thể thay đổi và đôi khi bị 404)
wget -O OpenJDK17U-jdk_x64_linux_hotspot.tar.gz \
  https://github.com/adoptium/temurin17-binaries/releases/latest/download/OpenJDK17U-jdk_x64_linux_hotspot.tar.gz

3) Giải nén JDK

tar -xzf OpenJDK17U-jdk_x64_linux_hotspot.tar.gz
ls -la /usr/java

Sau khi giải nén, thường sẽ ra thư mục dạng jdk-17.0.x+....

7) Nếu dùng cho WildFly chạy bằng systemd (khuyến nghị set riêng trong service 7.3.4.C)

Với WildFly chạy dưới systemd, nên khai báo JAVA_HOME trong file service (hoặc override) để không phụ thuộc việc user/session có load /etc/profile hay không.

7.3.4) Chạy nhiều instance WildFly (multi-instance) trên CentOS 7 + DirectAdmin + OpenLiteSpeed

Mục tiêu:

  • Mỗi app chạy một instance riêng (thư mục riêng dưới /opt/wildfly/).
  • Mỗi instance có port offset riêng để không đụng port.
  • Quản lý start/stop/restart bằng systemd.
  • WildFly chỉ bind nội bộ 127.0.0.1 để không public ra internet (OLS/DA sẽ reverse proxy ở mục 7.3.6).

1) Cấu trúc thư mục

Sau khi cài WildFly (symlink /opt/wildfly đã có ở 7.3.3):

/opt/wildfly
├── bin
├── modules
├── standalone
├── <INSTANCE_NAME_1>
├── <INSTANCE_NAME_2>
└── ...

Trong đó, mỗi instance (<INSTANCE_NAME_1>, <INSTANCE_NAME_2>...) là bản copy của thư mục standalone.

Ví dụ:

/opt/wildfly
├── bin
├── modules
├── standalone
├── masion-art
└── site2

2) Tạo instance (copy từ standalone)

Câu lệnh tổng quát:

cp -a /opt/wildfly/standalone /opt/wildfly/<INSTANCE_NAME>
chown -R wildfly:wildfly /opt/wildfly/<INSTANCE_NAME>

Ví dụ (INSTANCE_NAME=masion-art):

cp -a /opt/wildfly/standalone /opt/wildfly/masion-art
chown -R wildfly:wildfly /opt/wildfly/masion-art

3) Tạo service systemd

Trên CentOS 7 (CentOS-7_DA-OpenLiteSpeed), có 2 cơ chế phổ biến để quản lý multi-instance WildFly bằng systemd.

3.A) THAM KHẢO Tạo service systemd cho từng service riêng (ví dụ: masion-art)

Ý tưởng: mỗi instance có 1 service file riêng: wildfly-masion-art.service.

3.A.1) Tạo file service cho instance masion-art

cat > /etc/systemd/system/wildfly-masion-art.service << 'EOF'
[Unit]
Description=WildFly Instance masion-art
After=network.target

[Service]
Type=simple
User=wildfly
Group=wildfly
WorkingDirectory=/opt/wildfly

# Java cho WildFly (khuyến nghị set rõ trong systemd để không phụ thuộc /etc/profile)
Environment=JAVA_HOME=/usr/java/jdk-17
Environment=PATH=/usr/java/jdk-17/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin

# Port offset cho instance này
Environment=PORT_OFFSET=0

ExecStart=/opt/wildfly/bin/standalone.sh \
	-b 127.0.0.1 \
	-bmanagement 127.0.0.1 \
	--server-config=standalone.xml \
	-Djboss.server.base.dir=/opt/wildfly/masion-art \
	-Djboss.socket.binding.port-offset=${PORT_OFFSET}

Restart=on-failure
RestartSec=5
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload

3.A.2) Enable + start instance

systemctl enable wildfly-masion-art
systemctl start wildfly-masion-art

systemctl status wildfly-masion-art --no-pager

3.A.3) Lệnh quản lý + xem log

systemctl restart wildfly-masion-art
journalctl -u wildfly-masion-art -f

3.B) ÁP DỤNG: Tạo service systemd dạng template (khuyến nghị khi nhiều instance)

Cách chuẩn gọn nhất trên CentOS 7 là dùng template unit: wildfly@.service.

  • Khi chạy: systemctl start wildfly@masion-art
  • %i sẽ tự nhận là masion-art.

3.B.1) Tạo file service template

Câu lệnh tổng quát:

cat > /etc/systemd/system/wildfly@.service << 'EOF'
[Unit]
Description=WildFly Instance %i
After=network.target

[Service]
Type=simple
User=wildfly
Group=wildfly
WorkingDirectory=/opt/wildfly

# Mỗi instance có 1 file env riêng để set PORT_OFFSET
EnvironmentFile=-/etc/sysconfig/wildfly-%i

# Java cho WildFly (khuyến nghị set rõ trong systemd để không phụ thuộc /etc/profile)
Environment=JAVA_HOME=/usr/java/jdk-17
Environment=PATH=/usr/java/jdk-17/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin

ExecStart=/opt/wildfly/bin/standalone.sh \
	-b 0.0.0.0 \
	-bmanagement 127.0.0.1 \
	--server-config=standalone.xml \
	-Djboss.server.base.dir=/opt/wildfly/%i \
	-Djboss.socket.binding.port-offset=${PORT_OFFSET}

Restart=on-failure
RestartSec=5
LimitNOFILE=65535

[Install]
WantedBy=multi-user.target
EOF

systemctl daemon-reload

3.B.2) Tạo file env cho từng instance (PORT_OFFSET)

Câu lệnh tổng quát:

cat > /etc/sysconfig/wildfly-<INSTANCE_NAME> << 'EOF'
PORT_OFFSET=<PORT_OFFSET>
EOF

Ví dụ:

  • masion-art offset = 0 (HTTP 8080, Management 9990)
cat > /etc/sysconfig/wildfly-masion-art << 'EOF'
PORT_OFFSET=0
EOF
  • site2 offset = 100 (HTTP 8180, Management 10090)
cat > /etc/sysconfig/wildfly-site2 << 'EOF'
PORT_OFFSET=100
EOF

4) Enable + start instance bằng systemctl (theo cơ chế template wildfly@.service)

Với cơ chế 3.B (template unit), SERVICE_NAME sẽ là wildfly@<INSTANCE_NAME>.

Câu lệnh tổng quát:

systemctl enable wildfly@<INSTANCE_NAME>
systemctl start wildfly@<INSTANCE_NAME>

systemctl status wildfly@<INSTANCE_NAME> --no-pager

Ví dụ (INSTANCE_NAME=masion-art):

systemctl enable wildfly@masion-art
systemctl start wildfly@masion-art

systemctl status wildfly@masion-art --no-pager

5) Các lệnh quản lý instance

Câu lệnh tổng quát:

systemctl start wildfly@<INSTANCE_NAME>
systemctl stop wildfly@<INSTANCE_NAME>
systemctl restart wildfly@<INSTANCE_NAME>
systemctl status wildfly@<INSTANCE_NAME> --no-pager

journalctl -u wildfly@<INSTANCE_NAME> -f

Ví dụ (INSTANCE_NAME=masion-art):

systemctl restart wildfly@masion-art
journalctl -u wildfly@masion-art -f

6) Port sau khi offset (tham khảo nhanh)

WildFly mặc định (offset = 0):

  • HTTP: 8080
  • Management: 9990

Ví dụ: PORT_OFFSET=100 (site2)

  • HTTP: 8180
  • Management: 10090

7) Kiểm tra nhanh từ chính VPS (khuyến nghị)

Câu lệnh tổng quát:

curl -I http://127.0.0.1:<HTTP_PORT>/

Ví dụ:

# instance offset=0
curl -I http://127.0.0.1:8080/

# instance offset=100

Ghi chú quan trọng

  • Nếu bạn muốn dùng kiểu service “mỗi instance 1 file .service” thì vẫn được, nhưng template wildfly@.service sẽ dễ bảo trì hơn (thêm instance mới không phải copy/paste cả file service).

7.3.5) Deploy file WAR lên WildFly (CentOS 7 + DirectAdmin + OpenLiteSpeed)

Mục tiêu của mục này:

  • Deploy file WAR lên đúng instance WildFly (multi-instance theo 7.3.4)
  • WildFly chạy nội bộ (127.0.0.1), không public port ra internet
  • OLS/DirectAdmin sẽ proxy ở mục 7.3.6

1) Copy file WAR vào thư mục deployments

cp /path/to/<APP_NAME> /opt/wildfly/<INSTANCE_NAME>/deployments/

# Ví dụ (deploy root)
cp /home/masion-art/app/260312/ROOT.war /opt/wildfly/masion-art/deployments/

2) Phân quyền đúng cho user chạy WildFly

chown wildfly:wildfly /opt/wildfly/<INSTANCE_NAME>/deployments/<APP_NAME>

# Ví dụ
chown wildfly:wildfly /opt/wildfly/masion-art/deployments/ROOT.war

Gợi ý thêm (tùy chọn):

  • Nếu bạn thường copy bằng root, có thể set quyền thư mục để khỏi quên:
chown -R wildfly:wildfly /opt/wildfly/<INSTANCE_NAME>/deployments

3) Restart instance WildFly (khuyến nghị để chắc chắn)

WildFly có auto-deploy khi thả WAR vào deployments/, nhưng trên môi trường production thường restart để:

  • Đảm bảo instance load lại deployment mới
  • Tránh tình trạng deploy "lưng chừng" nếu đang có file copy dở hoặc app đang giữ resource
systemctl restart <SERVICE_NAME>

# Ví dụ
systemctl restart wildfly@masion-art

4) Theo dõi log deploy của instance (server.log)

tail -100f /opt/wildfly/<INSTANCE_NAME>/log/server.log

# Ví dụ
tail -100f /opt/wildfly/masion-art/log/server.log

5) Theo dõi log của service systemd

journalctl -u <SERVICE_NAME> -f

# Ví dụ
journalctl -u wildfly@masion-art -f

Tóm tắt ngắn:

  • Copy WAR vào: /opt/wildfly/<INSTANCE_NAME>/deployments/
  • Set owner: wildfly:wildfly
  • Restart service: wildfly@<INSTANCE_NAME>

Câu lệnh tổng quát:

cp /path/to/<APP_NAME> /opt/wildfly/<INSTANCE_NAME>/deployments/
chown wildfly:wildfly /opt/wildfly/<INSTANCE_NAME>/deployments/<APP_NAME>
systemctl restart wildfly@<INSTANCE_NAME>

Ví dụ (INSTANCE_NAME=masion-art, APP_NAME=ROOT.war):

cp /path/to/ROOT.war /opt/wildfly/masion-art/deployments/
chown wildfly:wildfly /opt/wildfly/masion-art/deployments/ROOT.war
systemctl restart wildfly@masion-art

7.3.6) Reverse proxy OpenLiteSpeed (OLS) trỏ về WildFly

Mục tiêu:

  • OLS nhận request public https://yourdomain.com/ (hoặc /api, /app)
  • Reverse proxy về WildFly chạy nội bộ (khuyến nghị bind 127.0.0.1)
  • Forward đúng header để WildFly:
    • Log đúng IP client
    • Hiểu request gốc là HTTPS
    • Tránh lỗi redirect sai protocol (http ↔ https)

A) Khai báo External App (OLS → WildFly)

Trên OLS WebAdmin (thường là https://<IP>:7080):

1) Vào Virtual Hosts → chọn vhost của domain

2) External App → Add

  • Type: Web Server
  • Name: wildfly_masion_art
  • Address: 127.0.0.1:8080
    • Nếu instance offset khác, đổi theo port nội bộ. Ví dụ site2 offset=100 → 127.0.0.1:8180
  • Max Connections: khuyến nghị 100–200 (tuỳ tải, RAM/CPU)
  • Keep Alive: Yes

Ghi chú nghiệp vụ:

  • Max Connections quá thấp dễ nghẽn khi có nhiều request đồng thời.
  • Keep Alive giúp giảm latency (đặc biệt với API nhiều request nhỏ) và giảm overhead tạo kết nối mới.

B) Tạo Proxy Context và bật header forwarding

3) Context → Add

  • Type: Proxy
  • URI:
    • Proxy toàn site: /
    • Hoặc proxy theo path: /api/ (tuỳ nhu cầu)
  • Location: wildfly_masion_art

4) Trong Proxy Context, bổ sung header forwarding (khuyến nghị):

  • X-Forwarded-For: để WildFly biết IP client (thay vì chỉ thấy 127.0.0.1)
  • X-Forwarded-Proto: để WildFly biết request gốc là https
  • X-Forwarded-Host: để WildFly biết đúng host/domain gốc
Tuỳ giao diện OLS/LSWS, phần này có thể nằm ở mục “Request Header” hoặc “Add/Set Header”. Nếu OLS cho phép “pass through” header sẵn có thì chỉ cần đảm bảo 3 header trên được set/forward.

C) Bật proxy-address-forwarding trong WildFly (BẮT BUỘC nếu dùng reverse proxy)

Trên instance WildFly, sửa file:

  • /opt/wildfly/<INSTANCE_NAME>/configuration/standalone.xml

Tìm http-listener (Undertow) và bật:

<http-listener name="default" socket-binding="http" proxy-address-forwarding="true"/>

Mục đích:

  • WildFly/Undertow tin và sử dụng các header X-Forwarded-* từ reverse proxy
  • Tránh lỗi redirect sai scheme (http/https)
  • Log đúng IP client khi app/framework đọc remote address

Sau khi sửa, restart instance:

systemctl restart wildfly@<INSTANCE_NAME>

D) Lưu ý quan trọng (case chạy root / và context)

  • Nếu muốn app chạy ở / để proxy đơn giản nhất, deploy dưới tên ROOT.war.
  • Khi đó, WildFly (offset 0) thường phục vụ tại http://127.0.0.1:8080/ và OLS context / sẽ hoạt động ổn định.
  • Nếu app chạy ở context /ten-app (ví dụ myapp.war), bạn có thể:
    • Proxy URI / và để app tự handle route, hoặc
    • Proxy URI /ten-app/ tuỳ cách bạn muốn expose.

E) Kiểm tra nhanh sau khi cấu hình

  • Test từ server (bỏ qua OLS):
curl -I http://127.0.0.1:8080/
  • Test qua domain:
curl -I https://yourdomain.com/
  • Nếu gặp redirect loop hoặc bị đá về http:
    • Kiểm tra proxy-address-forwarding="true"
    • Kiểm tra X-Forwarded-Proto=https có được forward/set đúng

7.3.7) Firewall cho WildFly

Không nên mở 8080 ra ngoài. Vì WildFly bind 127.0.0.1 nên mặc định cũng không bị public.

Nếu bạn đổi bind ra 0.0.0.0 thì phải chặn firewall hoặc chỉ allow IP nội bộ.

7.3.8) Log và debug nhanh

  • Xem log WildFly (đúng theo service instance):
journalctl -u wildfly@masion-art -n 200 --no-pager
  • Kiểm tra app từ server:
curl -I http://127.0.0.1:8080/
curl -I http://127.0.0.1:8080/myapp/
  • Log OLS:
    • tail -f /usr/local/lsws/logs/error.log
    • Access log theo vhost.

8) Cài database (MySQL) và hardening cơ bản

Mục tiêu của phần này:

  • Cài/kiểm tra MySQL hoạt động ổn định (service, port, version)
  • Hardening nhanh bằng mysql_secure_installation theo chuẩn production
  • Tạo database + user cho application theo nguyên tắc least privilege

8.1) Kiểm tra MySQL đã có chưa

systemctl status mysqld || systemctl status mysql
mysql --version
ss -tulnp | grep 3306 || true

# Kiểm tra password root DirectAdmin
# DirectAdmin thường lưu password tại: DirectAdmin thường lưu password tại:
Lệnh: 
	cat /usr/local/directadmin/conf/mysql.conf
	hoặc: cat /root/.my.cnf

Gợi ý đọc nhanh:

  • Nếu ss không ra gì: service chưa chạy, hoặc MySQL chưa cài.
  • Nếu MySQL chỉ dùng nội bộ cho web/app, không cần expose 3306 ra internet.

8.2) Hardening database với mysql_secure_installation (khuyến nghị)

[Bỏ qua vì không có tài khoản root]

Chạy lệnh:

mysql_secure_installation

Script sẽ hỏi các câu sau, nên trả lời theo cấu hình chuẩn production:

  • Set root password? → Y (Phải check hiện tại user root đã có pass chưa?)
  • Remove anonymous users? → Y
  • Disallow root login remotely? → Y
  • Remove test database and access to it? → Y
  • Reload privilege tables now? → Y

Mục đích của từng tuỳ chọn (vì sao nên chọn Y):

  • Remove anonymous users: tránh login DB không có password.
  • Disallow root login remotely: giảm rủi ro bị dò password/chiếm quyền root từ xa.
  • Remove test database: DB test thường là mục tiêu bị exploit.
  • Reload privileges: áp dụng ngay thay đổi quyền sau khi hardening.

8.3) Tạo database + user cho app (ví dụ chuẩn)

Login MySQL:

mysql -u da_admin -p

jER0mmx3i084s

Tạo database và user chỉ được truy cập từ localhost:

DROP DATABASE IF EXISTS `ART_AND_DECOR`;

CREATE DATABASE IF NOT EXISTS `ART_AND_DECOR`
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

CREATE USER 'masion-art'@'localhost'
IDENTIFIED BY 'masion-art@123';

Kiểm tra nhanh user/host và quyền:

8.4) MySQL / MariaDB CLI cheat sheet (Linux VPS)

1) Login database

  • Login root:
mysql -u root -p
  • Logout:
exit;
  • Login user vào database cụ thể:
mysql -u <USERNAME> -p <DATABASE_NAME>

# Ví dụ
mysql -u masion-art -p ART_AND_DECOR
  • Login qua host:
mysql -u <USERNAME> -p -h localhost
  • Login qua IP:
mysql -u <USERNAME> -p -h 103.166.182.45

2) Các lệnh cơ bản trong MySQL

  • Xem danh sách database:
SHOW DATABASES;
  • Chọn database:
USE <DATABASE_NAME>;

# Ví dụ
USE ART_AND_DECOR;
  • Xem danh sách tables:
SHOW TABLES;
  • Xem cấu trúc table:
DESCRIBE users;

hoặc

SHOW CREATE TABLE users;
  • Xem dữ liệu:
SELECT * FROM users;
  • Giới hạn dữ liệu:
SELECT * FROM users LIMIT 10;

3) Chạy file SQL

Cách 1: chạy từ terminal

  • Import 1 database:
mysql -u <USERNAME> -p <DATABASE_NAME> < <FILE_NAME.sql>

# Ví dụ
mysql -u masion-art -p ART_AND_DECOR < RUN_ALL_DATABASE_SCRIPTS.sql;
masion-art@123

Cách 2: chạy từ bên trong MySQL CLI

SOURCE <FILE_NAME.sql>;

hoặc

SOURCE /full/path/to/file.sql;

Ví dụ:

SOURCE /home/admin/backup.sql;

4) Backup database

  • Backup 1 database:
mysqldump -u root -p app_db > backup.sql
  • Backup tất cả database:
mysqldump -u <USERNAME> -p --all-databases > alldb.sql
  • Backup chỉ schema (structure):
mysqldump -u root -p --no-data app_db > schema.sql
  • Backup chỉ dữ liệu:
mysqldump -u root -p --no-create-info app_db > data.sql

5) Tạo database

CREATE DATABASE app_db
CHARACTER SET utf8mb4
COLLATE utf8mb4_unicode_ci;

6) Tạo user database

CREATE USER 'app_user'@'localhost'
IDENTIFIED BY 'strong_password';

7) Cấp quyền user

ON app_db.*
TO 'app_user'@'localhost';

8) Reload quyền

FLUSH PRIVILEGES;

9) Xoá database

DROP DATABASE app_db;

10) Xoá user

DROP USER 'app_user'@'localhost';

11) Kiểm tra user database

SELECT user,host FROM mysql.user;

12) Kiểm tra quyền user

SHOW GRANTS FOR 'app_user'@'localhost';

13) Kiểm tra database đang sử dụng

SELECT DATABASE();

14) Kiểm tra version database

SELECT VERSION();

Hoặc ngoài terminal:

mysql --version

15) Kiểm tra port database

systemctl status mariadb

hoặc

systemctl status mysql

17) Restart database

systemctl restart mariadb

18) Kiểm tra connection database

SHOW PROCESSLIST;

19) Kill connection

20) Kiểm tra dung lượng database

	table_schema AS "Database",
	ROUND(SUM(data_length + index_length) / 1024 / 1024, 2) AS "Size (MB)"
FROM information_schema.tables
GROUP BY table_schema;

10) Đánh giá nhanh tài liệu nguồn và điều chỉnh cho CentOS 7 + DA + OLS

Từ tài liệu “deploy server VPS Linux” trước đó:

  • Phần giải thích aaPanel/DirectAdmin vẫn đúng, nhưng triển khai mới sẽ chuẩn hoá theo DirectAdmin + OpenLiteSpeed.
  • Các phần về Nginx reverse proxy và mẫu config Nginx sẽ không áp dụng trực tiếp khi dùng OLS (OLS có cơ chế reverse proxy riêng).
  • Phần Certbot + Nginx plugin không phù hợp. Với DA + OLS, nên dùng Let’s Encrypt tích hợp trong DirectAdmin để tránh lệch cấu hình.
  • Các ví dụ dùng dnf (Rocky/Alma/CentOS 8+) cần đổi sang yum trên CentOS 7.

Checklist triển khai nhanh (tóm tắt)

  • Update server + cài tool cơ bản
  • Thiết lập hostname + DNS
  • Bật firewalld và mở port 22/80/443/2222
  • Cài DirectAdmin
  • Chọn/đổi sang OpenLiteSpeed trong CustomBuild
  • Tạo domain + bật Let’s Encrypt
  • Deploy website/app + kiểm tra log
